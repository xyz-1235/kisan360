UNIT 4A Deadlocks ▾
📌 What is a Deadlock?
  • A set of processes is deadlocked when every process in the set is waiting for an event that can only be caused by another process in the same set — and no process can ever proceed.
  • Resources can be physical (printers, tape drives, memory) or logical (files, semaphores, monitors).
  • Normal resource use sequence: Request → Use → Release.
  • System maintains a table recording whether each resource is free or allocated and to which process.
🔒 4 Necessary Conditions (Coffman Conditions)

ALL four must hold simultaneously for a deadlock to occur.

#ConditionMeaning
1Mutual ExclusionAt least one resource held in non-sharable mode; only one process can use it at a time.
2Hold and WaitA process holds at least one resource and waits for additional resources held by others.
3No PreemptionResources cannot be forcibly taken; released only voluntarily after task completion.
4Circular WaitA circular chain P0→P1→…→Pn→P0 exists where each waits for a resource held by the next.
📊 Resource Allocation Graph (RAG)
  • Vertices: Processes (circles) and Resources (rectangles with dots for instances).
  • Request edge: Pi → Rj (process requesting resource — arrow FROM process TO resource).
  • Assignment edge: Rj → Pi (resource assigned to process — arrow FROM resource TO process).
  • No cycle in graph → No deadlock.
  • Cycle exists, each resource has ONE instance → Deadlock guaranteed (necessary AND sufficient).
  • Cycle exists, resources have MULTIPLE instances → Deadlock may or may not exist (only necessary condition).
KEY: If P4 can release a resource and break the cycle → no deadlock. Cycle alone ≠ deadlock when multiple instances exist.
🛡️ Deadlock Prevention (Attack each condition)

🧭 Deadlock Avoidance — Safe State
  • Safe State: System can allocate resources to each process in some order (safe sequence) and avoid deadlock.
  • Safe sequence <P1, P2, ..., Pn>: for each Pi, its remaining requests can be satisfied by currently available resources PLUS resources held by all Pj (j < i).
  • Safe state → NOT deadlocked. Deadlocked state → Unsafe. But not all unsafe states lead to deadlock.
  • Avoidance strategy: grant a request ONLY if the allocation leaves the system in a safe state.
Unsafe state example: 12 tape drives, P0 needs 10 (has 5), P1 needs 4 (has 2), P2 needs 9 (has 2). At t0 = safe. If P2 gets 1 more → only 2 free → only P1 can complete → 4 free → P0 needs 5 more but only 4 → UNSAFE!

Resource-Allocation Graph Algorithm (Single instance per resource)

  • Adds a claim edge (dashed) Pi → Rj: process Pi may request Rj in future.
  • Claim edge converts to request edge when process actually requests.
  • Request granted only if converting request edge → assignment edge does NOT create a cycle.
  • Cycle detection requires O(n²) operations.

Banker's Algorithm (Multiple instances per resource)

  • Each process must declare maximum need in advance.
  • On each request, simulate allocation; if resulting state is safe → grant; else → wait.
  • Uses: Available, Max, Allocation, Need = Max − Allocation matrices.
🔧 Methods for Handling Deadlocks (3 Approaches)
  • Prevention/Avoidance: Ensure system never enters deadlock state.
  • Detection + Recovery: Allow deadlock, detect it, recover from it.
  • Ignore (Ostrich Algorithm): Used by most OSes (UNIX, Windows). Left to application developer. Most practical for low-frequency deadlocks.
🚑 Recovery from Deadlock

Process Termination

  • Abort ALL deadlocked processes — breaks cycle; expensive (work lost).
  • Abort ONE at a time — run detection after each abort; overhead heavy.
  • Selection criteria: priority, compute time, resources held, resources still needed, number of processes to terminate, interactive vs. batch.

Resource Preemption

  • Select victim — minimize cost (resources held, time consumed).
  • Rollback — roll back to safe state; simplest = total rollback (abort & restart).
  • Starvation — same process repeatedly picked as victim; solution: include rollback count in cost factor.
UNIT 4B Memory Management — Basic Concepts ▾
🖥️ Basic Hardware
  • Only main memory and CPU registers are directly accessible by the CPU. Data not in memory must be moved there first.
  • Cache memory buffers speed difference between CPU and main memory.
  • Each process has a separate memory space, protected by base and limit registers.
  • Base register = smallest legal physical address. Limit register = size of the range.
  • CPU hardware compares every user-mode address with these registers; violation → trap (fatal error).
  • Only OS (in kernel mode) can load the base and limit registers (privileged instruction).
Legal range: Base ≤ address ≤ Base + Limit Example: Base = 300040, Limit = 120900 → legal addresses: 300040 to 420939
📍 Address Binding

A program goes through stages before execution. Addresses change form at each stage:

  • Source program: symbolic addresses (e.g., count)
  • Compiler: relocatable addresses (e.g., "14 bytes from module start")
  • Linker/Loader: absolute addresses (e.g., 74014)
Binding TimeWhenCharacteristic
Compile TimeIf memory location known at compile timeGenerates absolute code; must recompile if location changes. (MS-DOS .COM files)
Load TimeLocation not known at compile timeGenerates relocatable code; final binding at load time; just reload on address change
Execution TimeProcess can move during executionBinding delayed to runtime; needs MMU hardware. Used by most modern OSes
🔄 Logical vs Physical Address Space
  • Logical address (virtual address): Generated by CPU. What program sees.
  • Physical address: Seen by memory unit; loaded into memory address register.
  • Compile-time and load-time binding → logical = physical.
  • Execution-time binding → logical ≠ physical (they differ!).
  • MMU (Memory Management Unit): Hardware device that maps virtual → physical at runtime.
  • Base register = relocation register. Physical address = logical address + relocation register value.
  • User program NEVER sees real physical addresses; always works with logical addresses.
Physical Address = Logical Address + Relocation Register Example: Base = 14000, Logical = 346 → Physical = 14346
📦 Dynamic Loading
  • A routine is not loaded until it is called. All routines kept on disk in relocatable format.
  • Main program loads; when routine needed, relocatable linking loader loads it and updates address tables.
  • Advantage: Unused routines never loaded → better memory utilization. Ideal for large programs with infrequently used error handlers.
  • Does NOT require special OS support — responsibility of programmer/user.
🔁 Swapping
  • A process can be swapped out to backing store (fast disk) and later swapped in for continued execution.
  • In round-robin scheduling: when quantum expires, memory manager swaps out finished process, swaps in next.
  • Roll out, roll in: Priority-based variant — higher-priority process preempts lower-priority; lower swapped out, later restored.
  • Normally process swaps back to same memory space (if compile/load-time binding). With execution-time binding, can swap to different location.
  • Backing store: Fast disk; must accommodate all memory images; system maintains a ready queue.
  • Standard swapping rarely used today — too slow. Modified versions used in UNIX (triggered by high memory load).
Dispatcher checks if next process is in memory. If not and no free memory → swaps out a current process first, then loads desired process.
UNIT 5 Memory Allocation, Paging, Segmentation & Virtual Memory ▾
🗂️ Contiguous Memory Allocation
  • Memory split into: OS partition + User process partition.
  • Each process occupies a single contiguous section of memory.
  • Relocation register: smallest physical address. Limit register: range of logical addresses.
  • Every address checked: logical address must be < limit; physical = logical + relocation.

💥 Fragmentation

External Fragmentation

  • Total memory space is enough but it is not contiguous.
  • Free space scattered in small holes between processes.
  • Suffered by: First Fit and Best Fit.
  • Solution: Compaction — shuffle memory to combine all holes. Only possible with execution-time binding.

Internal Fragmentation

  • Memory allocated to a process is slightly larger than requested.
  • Difference between allocated and actually needed = wasted space inside the partition.
  • Caused by: fixed block allocation (paging).
📄 Paging — Basic Method
  • Physical memory split into fixed-size blocks called frames. Logical memory split into same-size blocks called pages.
  • When process executes, its pages loaded into any available frames (non-contiguous OK!).
  • CPU generates: page number (p) + page offset (d).
  • Page table: maps page number → frame number (base address in physical memory).
  • Physical address = (frame number × page size) + offset.
  • Paging eliminates external fragmentation but may cause internal fragmentation (last frame may not be full).
  • Page size is power of 2 (typically 512B to 16MB) — makes translation easy.
Logical address size = 2ᵐ bits, Page size = 2ⁿ → High-order (m−n) bits = page number | Low-order n bits = page offset Example: Page size 4B, logical addr 13 → page 3 (=13÷4), offset 1 (=13%4) If page 3 → frame 2 → Physical = 2×4 + 1 = 9

Frame Table

  • OS maintains a frame table: one entry per physical frame, shows free/allocated and which process's page it holds.
  • OS keeps copy of page table for address translation and for CPU dispatcher to set hardware page table.
⚡ TLB — Translation Lookaside Buffer
  • Page table in main memory requires 2 memory accesses (page table + actual data) → slow!
  • TLB: Small, fast associative cache. Stores subset of page table entries.
  • TLB search: Page number compared with ALL keys simultaneously.
  • TLB Hit: Frame number immediately available → 1 memory access total.
  • TLB Miss: Access page table in memory → get frame, then add (page#, frame#) to TLB → 2 memory accesses.
  • TLB typically holds 64–1024 entries. Replacement: LRU or random.
  • Hit ratio: % of times page number found in TLB.
  • ASID (Address Space ID): Stored in TLB entries to identify processes; prevents wrong translations during context switch. Without ASID → flush TLB on every context switch.

Protection in Paging

  • Each page table entry has protection bits (read-write / read-only) and a valid-invalid bit.
  • Valid bit = page in process's logical address space. Invalid = page not in address space (trap if accessed).
📏 Segmentation
  • Supports user's view of memory as a collection of variable-sized named segments (code, stack, heap, global data, etc.).
  • Logical address: <segment-number s, offset d>
  • Segment table: each entry has base (starting physical address) and limit (length of segment).
  • If d < limit: physical address = base + d. Else: trap (addressing error).
  • Eliminates internal fragmentation but suffers from external fragmentation.

🌐 Virtual Memory — Background
  • Allows execution of processes not completely in memory. Programs can be larger than physical memory.
  • Abstracts main memory into a large uniform array of storage — separates logical from physical memory.
  • Benefits: Programs not constrained by physical memory size; more processes run simultaneously; less I/O needed.
  • Virtual address space: starts at 0, contiguous — heap grows up, stack grows down. Space between = sparse (hole).
  • Virtual memory also enables shared libraries, shared memory between processes, and faster process creation via fork().
📥 Demand Paging
  • Load pages only when needed (demanded). Pages never accessed → never loaded.
  • Uses a lazy swapper (pager) — never swaps a page in unless needed. Pager handles individual pages; swapper handles entire processes.
  • Uses valid-invalid bit: valid = page in memory; invalid = page on disk or not in address space.
  • Process runs normally until it accesses an invalid page → Page Fault!

Page Fault Handling Steps

  • 1. Check internal table (PCB) — valid or invalid reference?
  • 2. Invalid reference → terminate process. Valid but not in memory → continue.
  • 3. Find a free frame from free-frame list.
  • 4. Schedule disk read of desired page into new frame.
  • 5. When disk read complete → update page table (mark valid).
  • 6. Restart the instruction that caused the trap.
Pure Demand Paging: Start process with NO pages in memory. Pages faulted in as needed. Theoretically valid; locality of reference makes it practical.
🔄 Page Replacement
  • When a page fault occurs and NO free frame exists → need to replace an existing page.
  • Modify (dirty) bit: If set, page has been modified → must write to disk before replacing. If not set → just overwrite (saves I/O).
  • Reference string evaluated to compare algorithms. Goal: minimize page faults.

FIFO Page Replacement Simple

  • Replace the page that has been in memory the longest.
  • Implemented with a queue; replace the head, insert new at tail.
  • For reference string 7,0,1,2,0,3,0,4,2,3,0,3,2,1,2,0,1,7,0,1 with 3 frames → 15 page faults.
  • Belady's Anomaly: MORE frames can lead to MORE page faults with FIFO (e.g., string 1,2,3,4,1,2,5,1,2,3,4,5 → 3 frames=9 faults, 4 frames=10 faults).

Optimal Page Replacement (OPT / MIN) Best

  • Replace the page that will NOT be used for the longest time in the future.
  • Guarantees the lowest possible page fault rate. Never suffers Belady's anomaly.
  • Same reference string → 9 page faults.
  • Impossible to implement in practice (requires future knowledge). Used as a benchmark.

LRU Page Replacement Most Used

  • Replace the page that has not been used for the longest time (past approximates future).
  • Associates each page with the time of its last use.
  • Same reference string → 12 page faults (between FIFO=15 and OPT=9).
  • Does NOT suffer from Belady's anomaly.
  • Implementation needs hardware assistance (counter or stack for each page).

UNIT 6 File System Management ▾
📁 File Concept
  • A file is a named collection of related information recorded on secondary storage. Smallest logical allotment of secondary storage.
  • Can represent programs (source/object) or data (numeric, text, binary).
  • Types by structure: text file, source file, object file, executable file.

File Attributes

  • Name: Only human-readable info. Identifier: unique number within file system.
  • Type: needed for systems supporting different file types. Location: pointer to device and location on it.
  • Size: current size in bytes/words/blocks. Protection: access-control info.
  • Time, date, user ID: for creation, last modification, last use.
⚙️ File Operations (6 Basic)
  • Create: Find space in file system + make directory entry.
  • Write: System call with file name + info to write. System maintains write pointer.
  • Read: System call with file name + memory location. System maintains read pointer.
  • Reposition (Seek): Move current-file-position pointer to given value. No actual I/O needed.
  • Delete: Find in directory, release file space, erase directory entry.
  • Truncate: Erase content but keep attributes; reset length to 0, release space.
open() system call: Searches directory, copies entry to open-file table, returns pointer (file descriptor) for subsequent I/O — avoids repeated directory searches.

Open-file table info: File pointer (current position), File-open count (how many processes), Disk location (cached), Access rights.

Mandatory locking (Windows): OS prevents access to locked file. Advisory locking (UNIX): Software must acquire lock manually.

🔍 File Access Methods

Sequential Access

  • Process data in order, one record after another.
  • read next: reads next portion, advances file pointer.
  • write next: appends to end, advances pointer.
  • Based on tape model. Most common access method.
  • Used by editors, compilers.

Direct Access (Relative)

  • Fixed-length logical records; read/write in any order.
  • File = numbered sequence of blocks/records.
  • Based on disk model (random access).
  • Operations: read n, write n, or position to n + read next.
  • Uses relative block number (index from file start).
  • Great for databases.

Index Sequential Access (ISAM)

  • Index file contains pointers to data blocks. Binary search on index → find block → sequential within block.
  • For large files: multilevel index (primary index → secondary index → data).
  • Allows both sequential and direct access.
🗃️ Directory Structure

Storage Structure

  • Disk divided into partitions/slices/volumes. Each can have its own file system.
  • Volume = virtual disk. Contains a device directory / volume table of contents with info (name, location, size, type) for all files.

1. Single-Level Directory

  • All files in ONE directory. Simple but limited.
  • Problems: All file names must be unique; difficult for multiple users; name collision inevitable.

2. Two-Level Directory

  • Master File Directory (MFD): indexed by user name/account; points to each user's directory.
  • User File Directory (UFD): each user has their own. Different users can have files with the same name.
  • Path: /username/filename. Problem: isolation — users can't share files easily.

3. Tree-Structured Directory

  • Arbitrary depth tree. Users create subdirectories. Most common structure.
  • Absolute path: from root to file. Relative path: from current directory.
  • Each process has a current directory. Change with system call.
  • Directory deletion: MS-DOS (must be empty) vs UNIX rm -r (recursive delete).
  • One bit per directory entry: 0 = file, 1 = subdirectory.

4. Acyclic-Graph Directory

  • Allows shared files/directories — same file can appear in two different directories.
  • Implemented via links (UNIX) — pointer (absolute/relative path) to actual file.
  • Shared file ≠ two copies. Changes by one user visible to all immediately.
  • Problem: dangling pointers if file deleted while links remain. Solution: reference counting (delete only when count = 0).
  • A file may have multiple absolute path names (aliasing problem).

5. General Graph Directory

  • Acyclic graph + links → may create cycles.
  • Cycles make traversal complex (infinite loops). Solution: limit number of directories accessed in search.
  • Reference count may not reach 0 with cycles → need garbage collection (mark-and-sweep), but very time-consuming for disks.
🔐 Protection

Types of Access

  • Read, Write, Execute, Append, Delete, List.
  • Other operations (rename, copy, edit) often implemented using lower-level calls.

Access Control List (ACL)

  • Associate each file with list of (user, access-type) pairs.
  • Problem: list can be very long. Solution: condensed ACL using three classes:
  • Owner (creator), Group (sharing users), Universe (all others).
  • UNIX: 3 fields × 3 bits = 9 bits per file. rwx rwx rwx (owner, group, others).
  • Solaris: combines three-category with ACL (ACL for fine-grained, ACL takes precedence).
  • Windows (NTFS): ACL management via GUI.
💾 Disk Space Allocation Methods

1. Contiguous Allocation Simple

  • Each file occupies a set of contiguous blocks on disk.
  • Directory entry stores: start block + length (in blocks).
  • File of n blocks starting at b: occupies blocks b, b+1, ..., b+n-1.
  • Supports both sequential and direct access efficiently (block i → access block b+i).
  • Problems: External fragmentation; must know file size at creation; extending a file is difficult (adjacent space may be taken).
  • Strategies for free space: First Fit, Best Fit. Both suffer external fragmentation.

2. Linked Allocation Flexible

  • Each file = linked list of disk blocks; blocks scattered anywhere on disk.
  • Directory stores: pointer to first and last block. Each block contains pointer to next.
  • No external fragmentation. File size need not be declared. Files grow as long as free blocks available.
  • Problems: Only efficient for sequential access (direct access requires traversal); pointer space overhead (4B per 512B block = 0.78%); reliability (corrupt pointer breaks chain).
  • Solution for pointer overhead: clusters (group blocks). Increases internal fragmentation.
  • FAT (File Allocation Table): Variation. Table at start of volume, one entry per block. Directory has first block number; FAT has chain of next blocks. Used by MS-DOS, OS/2. Improves random access but requires FAT caching.

3. Indexed Allocation Best for Direct

  • All block pointers collected into one index block per file.
  • Directory stores address of the index block. ith entry in index block → ith block of file.
  • Supports direct access without external fragmentation.
  • Problems: Index block takes space even for small files. Pointer overhead greater than linked allocation.
  • For very large files: multilevel index (linked index blocks or combined scheme).
